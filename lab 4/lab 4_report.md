# Продуктивное решение
Продуктовое решение (Production) представляет собой высоконагруженную архитектуру, рассчитанную на 3 миллиона диалогов в месяц с требованиями SLA 99.9% и выше. Основой вычислений выступают выделенные ресурсы с годовым контрактом (CUD), что дает до 50% экономии по сравнению с pay-as-you-go моделью. Для обеспечения глобальной доступности используются Cloud CDN для кэширования статики, Cloud Armor для защиты от DDoS-атак и глобальный балансировщик нагрузки. Высокопроизводительная база данных реализована на AlloyDB (или Cloud Spanner), обеспечивающей 99.99% SLA и автоматическое масштабирование хранилища до 128 ТБ. Кэширование контекста выполняется через Memorystore (Redis), что снижает затраты на повторяющиеся токены. AI-модель запускается на выделенных GPU/TPU через Vertex AI с использованием Committed Use Discounts, что гарантирует стабильную латентность. Оркестрация микросервисов может выполняться как через Cloud Run с авто-масштабированием, так и через GKE Autopilot для более сложных сценариев. Такая архитектура обеспечивает отказоустойчивость, высокую производительность и оптимальное соотношение цены и качества при больших объемах трафика.
5.1 Почему Serverless (Cloud Run) на MVP и Тестировании. Cloud Run масштабируется до нуля, исключая плату за простой, что критически важно для MVP с непредсказуемой нагрузкой. Модель оплаты только за фактические запросы минимизирует расходы на ранних этапах. Сервис входит в бесплатный тариф Always Free (2 млн запросов/мес), делая MVP практически бесплатным. Serverless-подход избавляет от администрирования инфраструктуры, сокращая время развертывания.

## 5. Обоснование выбора ресурсов

5.2 Почему Cloud SQL вместо Firestore. Логи диалогов представляют собой структурированные данные, которые удобнее анализировать с помощью SQL-запросов для последующего дообучения модели (fine-tuning). Cloud SQL поддерживает ACID-транзакции и сложные JOIN-операции. При нагрузке до 300 тыс. диалогов/мес Cloud SQL (~$15) дешевле Firestore. Встроенные автоматические бэкапы обеспечивают сохранность данных при экспериментах с партнерами.

5.3 Почему Cloud Tasks. Очередь задач обеспечивает асинхронную обработку: при пиковых нагрузках запросы не теряются, а обрабатываются равномерно. Встроенные механизмы повторов автоматически перезапускают упавшие задачи, повышая надежность интеграции с партнерами. Возможность ограничения скорости (rate limiting) защищает бэкенд от перегрузок. Стоимость сервиса крайне низка (~$0.40 за 1 млн задач).

5.4 Почему в проде — Provisioned ресурсы. При 3 млн диалогов/мес Pay-as-you-go модель становится невыгодной: разница с выделенными ресурсами составляет $2,000+ в пользу Provisioned. Годовые контракты (CUD) дают скидку до 50%. Выделенные GPU/TPU обеспечивают стабильную латентность, необходимую для выполнения SLA с корпоративными партнерами. Позволяют точно планировать бюджет на длительный срок.

5.5 Почему AlloyDB, а не Cloud SQL. AlloyDB в 2-4 раза быстрее Cloud SQL для аналитических запросов. Поддерживает автоматическое масштабирование хранилища до 128 ТБ без простоя. Обеспечивает 99.99% SLA против 99.95% у Cloud SQL, что критично для продакшена. Интегрируется с Vertex AI для работы с векторными представлениями (эмбеддингами) для семантического поиска и рекомендаций. При доле затрат на БД в общем бюджете продакшена около 10%, разница в производительности оправдывает более высокую стоимость.
